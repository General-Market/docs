# Vision (Prediction Market)

Vision is a sealed parimutuel prediction market built on the General Market L3 chain. Traders bet on whether asset prices move UP or DOWN over fixed time intervals called ticks. All bets within a tick share a common pool — winners split the losers' stakes, minus a 0.3% protocol fee on profits.

<Info>
Vision uses the internal technical name "P2Pool" in contract code and API endpoints. User-facing references use "Vision."
</Info>

## Batches

A batch is a group of markets with a shared pool, tick duration, and resolution configuration. Any user can create a batch by selecting:

- **Markets** — 1 to 100+ assets (e.g., BTC, ETH, SOL)
- **Tick duration** — the time interval between resolutions: 5min, 10min, 30min, 1h, 4h, or 1 day
- **Resolution types** — how each market determines winners (see [Resolution Types](/reference/resolution-types))

Once created, a batch runs continuously. New players can join at any time by depositing USDC and submitting bet commitments.

## Bitmaps

A bitmap is a packed binary representation of a player's bets across all markets in a batch. Each market gets exactly 1 bit:

- **1 = UP** — the player bets the asset price will increase
- **0 = DOWN** — the player bets the asset price will decrease

Bitmaps use big-endian bit ordering: bit 0 is the MSB of byte 0, bit 7 is the LSB of byte 0, bit 8 is the MSB of byte 1, and so on. The total bitmap size is `ceil(marketCount / 8)` bytes.

### Sealed Commitment

Bitmaps follow a commit-reveal scheme:

1. **Commit** — the player submits `keccak256(bitmap)` on-chain when joining the batch. The hash is stored in the Vision contract.
2. **Reveal** — the player sends the actual bitmap bytes to issuer nodes via `POST /p2pool/bitmap`. Issuers verify the hash matches the on-chain commitment.

The sealed commitment prevents front-running: other players cannot see bet directions until after the commitment deadline.

## Ticks

A tick is one resolution interval within a batch. At the end of each tick:

1. Issuer nodes fetch closing prices for all markets in the batch
2. Each market resolves to a binary outcome based on price movement and the configured resolution type
3. Each player's bitmap is compared against the actual outcomes
4. The tick pool (sum of all players' `stakePerTick`) is distributed to winners proportionally
5. Losers forfeit their `stakePerTick` for that tick

Ticks repeat automatically at the configured duration (e.g., every 1 hour) for as long as the batch remains active.

## Resolution Types

Each market in a batch has a resolution type that determines the price movement threshold for an UP outcome. See the full reference at [Resolution Types](/reference/resolution-types).

| Type | Threshold | UP wins when... |
|------|-----------|-----------------|
| UP_0 | 0% | Price increases by any amount |
| UP_30 | 0.30% | Price increases by at least 30 basis points |
| UP_X | Custom | Price increases by at least X basis points |
| DOWN_0 | 0% | Price decreases by any amount |
| DOWN_30 | 0.30% | Price decreases by at least 30 basis points |
| DOWN_X | Custom | Price decreases by at least X basis points |
| FLAT_0 | 0% | Price does not change |
| FLAT_X | Custom | Price stays within X basis points of open |

## Balance Proofs

Player balances are tracked off-chain by issuer nodes and verified on-chain via BLS signatures. To claim rewards or withdraw funds, a player requests a **balance proof** from the issuer network:

1. Player calls `GET /p2pool/balance/{batchId}/{player}`
2. Issuer nodes compute the player's current balance (deposits + winnings - losses - fees)
3. Issuers produce a BLS-signed message containing the balance
4. Player submits the balance and BLS signature to the Vision contract
5. The contract verifies the BLS signature against the registered issuer keys and updates the player's on-chain balance

<Warning>
BLS verification is never bypassed in Vision. Every claim and withdrawal requires a valid aggregated BLS signature from the issuer network — no test modes, no admin overrides.
</Warning>

## Fee Structure

Vision charges a **0.3% fee (30 basis points) on profits only**. Players who lose a tick pay no fee. The fee is deducted from winnings before payout.

```
fee = (winnings * 30) / 10000
payout = winnings - fee
```

Accumulated fees are collected by the protocol via `collectFees()` on the Vision contract.

## Smart Contract

The Vision contract (`Vision.sol`) manages all on-chain state:

| Function | Description |
|----------|-------------|
| `createBatch()` | Create a new batch with markets, resolution types, and tick duration |
| `joinBatch()` | Join a batch with USDC deposit, stake per tick, and bitmap hash commitment |
| `updateBitmap()` | Update bet commitment without re-joining |
| `deposit()` | Add USDC to an existing position |
| `claimRewards()` | Claim winnings for a tick range with BLS balance proof |
| `withdraw()` | Exit a batch completely with BLS-signed final balance |
| `registerBot()` | Register an automated trading bot (free, no collateral) |
