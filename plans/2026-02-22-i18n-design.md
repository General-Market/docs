# i18n Design — Korean, Japanese, Chinese Translation

**Date:** 2026-02-22
**Status:** Approved

## Summary

Full internationalization of the frontend with Korean (ko), Japanese (ja), and Simplified Chinese (zh) translations. Optimized for AI agent navigation and human usability.

## Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| URL routing | Subdirectory (`/ko`, `/ja`, `/zh`) | Best for SEO + AI agent navigation (deterministic, no JS required) |
| Library | `next-intl` | Purpose-built for Next.js 15 App Router, server component support |
| Chinese variant | Simplified (zh-CN) | Largest Chinese-speaking crypto market |
| Translation files | One JSON per locale per namespace | Agent-friendly: targeted edits to small files |
| DeFi terms | Hybrid — keep acronyms, translate descriptions | e.g. "NAV (순자산가치)" |
| Geo detection | Cloudflare CF-IPCountry (primary) → Vercel request.geo (fallback) | Already using Cloudflare + Vercel |
| Persistence | Cookie (NEXT_LOCALE) → Geo → Accept-Language → English | Works server-side, agent-friendly, no JS needed |

## Architecture

### Routing

```
app/
├── [locale]/
│   ├── layout.tsx      (sets html lang, loads messages, NextIntlClientProvider)
│   ├── page.tsx        (home)
│   ├── itp/[itpId]/
│   ├── vision/
│   ├── privacy/
│   └── terms/
├── api/                (no translation)
└── not-found.tsx
```

### Middleware Flow

```
Request → middleware.ts
  1. Path already has locale? → pass through
  2. Cookie NEXT_LOCALE set? → redirect to /{cookie_locale}/...
  3. CF-IPCountry header? → map country → redirect
  4. Vercel request.geo? → map country → redirect
  5. Accept-Language? → match supported locale → redirect
  6. Default → /en/...
```

Country mapping:
- KR → ko
- JP → ja
- CN, SG, MY → zh
- Everything else → en

### Translation Files

```
messages/
├── en/
│   ├── common.json       (nav, footer, buttons, errors, language switcher)
│   ├── markets.json      (market categories, labels)
│   ├── portfolio.json    (positions, orders, statuses)
│   ├── create-itp.json   (form labels, placeholders, validation)
│   ├── buy-modal.json    (transaction steps, micro-labels)
│   ├── sell-modal.json
│   ├── lending.json      (vault, borrow, repay, collateral)
│   ├── p2pool.json       (batches, leaderboard)
│   ├── backtest.json     (simulation labels)
│   ├── system.json       (status section)
│   └── seo.json          (metadata, descriptions, OG)
├── ko/  (same structure)
├── ja/  (same structure)
└── zh/  (same structure)
```

### Component Integration

**Server components:** `getTranslations()` from `next-intl/server`
**Client components:** `useTranslations('namespace')` hook
**Language switcher:** Dropdown in Header, sets cookie + navigates to new locale path
**SEO:** `generateMetadata()` per page with localized title/description + `hreflang` alternates
**Sitemap:** Generates entries for all locale/page combinations

### Hybrid Terms Convention

DeFi acronyms stay English, descriptions translated:
```json
{
  "nav_label": "NAV (순자산가치)",
  "apy_label": "APY (연수익률)",
  "tvl_label": "TVL (총예치금액)"
}
```

## Upgradeability

**Adding a new language:**
1. Add locale code to `SUPPORTED_LOCALES`
2. Add country mapping in middleware
3. Copy `messages/en/` → `messages/{new}/` and translate

No code changes — fully data-driven.

**Agent workflow:**
1. English files (`messages/en/`) are source of truth
2. Add new keys to `en/` first, then propagate to other locales
3. Key names are descriptive and hierarchical
4. `scripts/check-translations.ts` validates completeness across locales
